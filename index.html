Multifactor · Solid2FA

Opt-in and message-handling flow

Solid2FA gives each user one dedicated number that receives the texts their accounts send. Non-sensitive notifications are forwarded to the user's real phone. One-time passcodes are shown only inside the app and are never sent over SMS. This page documents how a user consents, how their phone is verified, how messages are handled, and how they opt out.

Prepared for Telnyx compliance Contact dave@multifactor.com Status early development, designed flow

System at a glance. Codes are held at the number and shown only in the app; only non-sensitive texts continue to the real phone.

What this guarantees

Affirmative opt-in first

No SMS reaches the real phone until the user explicitly activates the service and sees exactly what they are agreeing to, including who is sending and how to stop.

One number per user

Each user gets a single dedicated number reused across their accounts, the way a normal mobile number works. It is not a pool of disposable per-account numbers.

Codes are never sent by SMS

One-time passcodes are shown only inside the app. Only non-sensitive notifications are relayed onward to the real phone.

Always reversible

Users can unlink their number in the app at any time, and STOP is honored on the SMS channel.

Two message paths

The dedicated numbers are provisioned through Telnyx, which delivers each inbound SMS into a secure enclave, an isolated, hardware-protected environment where the message is processed so its contents are never exposed outside that boundary. Every inbound message is reviewed and classified inside the enclave before anything leaves it, and that classification decides where it goes: sensitive codes are routed to the app, and non-sensitive messages are forwarded to the phone.

One-time passcode

sensitive · routed to the app

A connected account sends a 2FA or OTP code to the dedicated number. Inside the secure enclave the message is reviewed and classified as sensitive, so it is routed to the app and is never relayed to the user's real phone.

Notification

non-sensitive · forwarded

A connected account sends a non-sensitive notification to the dedicated number. Inside the secure enclave the message is reviewed and classified as non-sensitive, so it is forwarded on to the user's real phone the way they normally receive it.

  TX->>App: Inbound message delivered to the app

On the roadmap

This is planned for a later release. We are including it so it can be considered in the messaging design now.

Agent-assisted number change

Planned

In a later release, an AI agent will set the dedicated number on the connected account for the user, so they do not have to do it manually. The user approves this per account, in advance, and the agent can only set the number. The agent runs outside the enclave, drives the account website with a headless browser, and uses a stored session rather than the user's credentials. The agent never holds the sensitive values itself: the secure enclave injects the new number, and any verification codes that are needed, directly into the account's form, while the agent only drives the navigation. The account first verifies on the user's existing phone, sending a confirmation code there that the user enters when the app prompts them, and the enclave injects that code. Some accounts then add a secondary step that sends a second code to the new dedicated number, which the enclave also injects. After the switch, the account's ongoing codes arrive at the dedicated number and stay in the enclave, which the agent never sees. Because every sensitive value is injected by the enclave and the agent only navigates, the agent never gains unauthorized control of the number or the user's codes.
